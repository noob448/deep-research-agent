# Deep Research Agent — 项目汇报 (V4.0)

## 一、项目目标

构建一个**可独立执行长程深度研究任务的多智能体系统**，输入一个课题，自动完成规划、多源搜索、交叉验证、结构化报告撰写和事实核验的全流程。

### 核心难点

| 难点 | 说明 | 解决思路 |
|------|------|---------|
| **上下文窗口爆炸** | 原始搜索结果动辄数千字，普通的单 Agent 会把上下文撑爆，导致信息丢失 | 每个 Researcher 独立上下文 → 只返回结构化摘要 |
| **维度遗漏** | 没有规划机制的 Agent"想到哪搜到哪"，容易漏掉关键维度 | Supervisor MECE 分解 → 互斥穷尽的子问题 |
| **多线程纵深矛盾** | 单 Agent 深入一个方向时会丢失其他方向的进展 | 多 Researcher 并行 + Supervisor 综合归档 |
| **幻觉无法追溯** | LLM 输出看起来合理但找不到出处，用户无法验证 | source_id 全程追踪 → 论断→来源→原文 可追溯 |
| **研究不可恢复** | 中途中断后无法继续，需要从头开始 | 独立 Run 目录 + events.jsonl 断点恢复 |
| **报告质量不可控** | 纯模型输出缺少质量把关环节 | Critic 审查 + Verifier 逐条核验 |

## 二、架构设计

### 整体拓扑

```
用户课题
   │
   ▼
┌─────────────────────────────────────────────────────┐
│ Supervisor (研究总监)                                  │
│ - MECE 分解课题为 N 个子问题                            │
│ - 编写结构化任务简报（目标/范围/工具优先级/预算）          │
│ - 并行委派 Researcher                                  │
│ - 写入 /notes/ 归档 → 查漏补缺 → 分节撰写 /report.md    │
│ - 自评 + 写 /research_summary.txt                      │
│ 工具: write_todos, task, write_file, read_file, ls      │
└──────┬──────────────────────────────────────────────┘
       │ task("researcher-N", 任务简报)
       ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ Researcher-1 │ │ Researcher-2 │ │ Researcher-3 │  ...动态1-5个
│ 独立上下文    │ │ 独立上下文    │ │ 独立上下文    │
│ OODA 循环    │ │ OODA 循环    │ │ OODA 循环    │
│ 搜索+抓取    │ │ 搜索+抓取    │ │ 搜索+抓取    │
└──────┬──────┘ └──────┬──────┘ └──────┬──────┘
       │ 结构化摘要     │ 结构化摘要     │ 结构化摘要
       └───────────────┼───────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ Critic (可选, max模式)                                │
│ - 读 /report.md + /notes/ → 评分 → 缺陷清单           │
│ - REQUIRES_REWORK: true/false                        │
│ 工具: read_file, ls（不搜索）                          │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ Verifier (事实核验)                                   │
│ - 抽取 report.md 中的 claims                          │
│ - 读取 /sources/<source_id>.txt 原始文件               │
│ - 逐条判定: SUPPORTED / PARTIAL / UNSUPPORTED          │
│ - 写入 claims.jsonl + verification_summary.md         │
└─────────────────────────────────────────────────────┘
```

### 为什么不让 Supervisor 自己搜索

如果 Supervisor 持有搜索工具，它会在规划阶段就开始搜索，导致三个问题：
1. 上下文过早膨胀，影响后续规划质量
2. 搜索结果和规划指令混在一起，模型容易混淆
3. 无法利用多 Researcher 的独立上下文实现信息隔离

**设计原则**：Supervisor 只编排不执行，Researcher 只执行不决策。

## 三、技术选型

### LLM 层

| 选项 | 选择 | 原因 |
|------|:--:|------|
| DeepSeek V4 Pro | ✅ | OpenAI 兼容协议，中文能力强，thinking 模式适合 Agent 任务 |
| GPT-4o | ❌ | 成本高，且本项目需要频繁 API 调用 |
| Claude API | ❌ | 国内网络不稳定，API 限流严格 |
| 本地部署 7B 模型 | ❌ | 推理能力不足，无法完成复杂的规划和多步搜索 |

**为什么不是"绑定 DeepSeek"**：项目使用 OpenAI 兼容协议，改 `AGENT_MODEL` 和 `DEEPSEEK_BASE_URL` 即可切换到任何厂商。

### 搜索引擎

| 选项 | 选择 | 原因 |
|------|:--:|------|
| DuckDuckGo | ✅ | 免费、无需 API Key、支持 site: 过滤器 |
| Tavily | ❌ | 需要付费 API Key，结果质量虽好但有成本 |
| Google Custom Search | ❌ | 每日配额限制，国内访问不稳定 |
| SerpAPI | ❌ | 付费，每月 50-100 次搜索不够用 |

### 学术搜索

| 选项 | 选择 | 原因 |
|------|:--:|------|
| OpenAlex | ✅ | 免费开放 API，2.4 亿篇论文，无需认证 |
| Crossref | ✅ | DOI/期刊/作者等元数据，与 OpenAlex 互补 |
| Semantic Scholar | ❌ | 虽有 API 但速率限制更严格 |
| arXiv API | ❌ | 仅覆盖预印本，范围太窄 |

### 重排模型

| 选项 | 选择 | 原因 |
|------|:--:|------|
| BGE-reranker-v2-m3 | ✅ | 开源、本地部署、中文友好，~300MB |
| Cohere Rerank | ❌ | 需要 API Key，有成本 |
| 不重排 | ❌ | DDGS 返回 10 条结果噪声多，直接给 LLM 浪费上下文 |

### Agent 框架

| 选项 | 选择 | 原因 |
|------|:--:|------|
| LangChain DeepAgents | ✅ | 内置 Planning/Offloading/Delegation/Memory 原语 |
| LangGraph 裸写 | ❌ | 需要手写 StateGraph，开发量大 |
| AutoGen/CrewAI | ❌ | 多 Agent 编排不如 DeepAgents 与 LangChain 的集成度 |
| 纯 Prompt 工程 | ❌ | 无法实现真正的上下文隔离 |

### 前端框架

| 选项 | 选择 | 原因 |
|------|:--:|------|
| React 19 + TypeScript | ✅ | 类型安全，生态成熟，适合数据密集型面板 |
| Vue 3 (旧版) | ❌ | 动态性足够但 TypeScript 支持不如 React |
| Streamlit | ❌ | 适合数据展示不适合 SSE 实时日志流 |

## 四、项目流程

### 7 阶段研究流水线

```
阶段 1: 制定计划
  ├─ write_todos 立即执行（铁律：第一个动作）
  ├─ 决定 Researcher 数量（1-5，根据课题复杂度）
  ├─ MECE 分解为互斥穷尽的子问题
  └─ 为每个子问题写结构化任务简报

阶段 2: (HITL 审批，可选)
  └─ request_plan_approval → 用户 approve/修改

阶段 3: 并行委托
  └─ 同一条消息中 task(researcher-1) + task(researcher-2) + ...

阶段 4: 归档查漏
  ├─ Researcher 返回 → write_file 写入 /notes/
  ├─ ls /notes 检查覆盖情况
  └─ 漏了 → 派定向补研究（不重派全部）

阶段 5: 分节撰写报告
  ├─ 读 /skills/academic-report/SKILL.md
  ├─ 第1节: 摘要 + 研究背景
  ├─ 第2节: 核心发现（含对比表）
  ├─ 第3节: 分析与讨论（矛盾标注、局限性）
  └─ 第4节: 结论 + 参考来源（编号制 [1][2]...）

阶段 6: (Critic 反思，可选)
  ├─ task("critic") → 读报告 → 评分 → 缺陷清单
  └─ REQUIRES_REWORK=true → 补研究 → 修订

阶段 7: 自我批判 + 归档
  ├─ 通读报告 → 自评弱点 → 标注证据薄弱处
  └─ 写 /research_summary.txt（200-500字浓缩摘要）
```

### 三级运行模式

| 模式 | 搜索上限 | 时间限制 | Critic | Verifier | 适用场景 |
|------|---------|---------|--------|----------|---------|
| ⚡ Fast | 5 次 | 5 分钟 | ❌ | ✅ | 简单事实查询 |
| 🔬 Deep | 20 次 | 无 | ❌ | ✅ | 常规深度研究（默认） |
| 🧠 Max | 20 次 | 无 | ✅ | ✅ | 复杂多维课题 |

## 五、技术细节

### 5.1 Researcher 的 OODA 循环

Researcher 不是"搜完所有预设关键词就停"，而是遵循 OODA（Observe-Orient-Decide-Act）循环：

```
[Observe]  已经搜到了什么？哪些 In-scope 点清楚了？哪些模糊？
[Orient]   还缺什么？该用哪个工具？
[Decide]   下一个具体、窄的查询是什么？
[Act]      调用工具，读结果
           → 回到 Observe，根据新信息决定下一步
```

**关键约束**：
- 搜索预算代码层硬拦截（超过直接返回"预算用尽"）
- 去重缓存按 Researcher 隔离（同一查询不重复）
- 上下文输出双重限制（18K 软警告 / 30K 硬停止）
- 强制充分性自评（每用满 1/3 预算时）

### 5.2 搜索质量管线

```
用户查询
  │
  ▼
DuckDuckGo / OpenAlex / Crossref / 百度学术
  │  多搜回 10 条原始结果
  ▼
Cross-Encoder 重排 (BGE-reranker-v2-m3)
  │  按 query→body 相关性精排，保留 top-4
  ▼
URL 规范化 + source_id 生成
  │  去除 utm_*/fbclid 等追踪参数，SHA-256 去重
  ▼
轻量返回给 Agent
  └─ source_id + snippet（≤1200 字符），全文自动保存到 /sources/
```

### 5.3 来源追踪体系

```
web_search  ─→ 注册 source_id（轻量，不保存全文）
web_fetch   ─→ 注册 source_id + 保存全文到 /sources/src_xxx.txt
search_openalex/crossref ─→ 注册 source_id + DOI/作者/年份元数据
search_cn   ─→ 注册 source_id（知乎/百科/学术）
  │
  ▼
sources.jsonl   ← 完整来源账本（元数据 + 原文路径）
  │
  ▼
claim_verifier.py
  ├─ 从 report.md 抽取 claims（找带引用标记 [N] 的句子）
  ├─ 每条 claim 绑定 source_ids
  ├─ 调 Verifier LLM 逐条核验（读取原始 source 文件）
  ├─ 输出: SUPPORTED / PARTIAL / UNSUPPORTED / CONTRADICTED
  └─ 写入 claims.jsonl + verification_summary.md
```

### 5.4 上下文控制策略

| 层级 | 机制 | 阈值 | 效果 |
|------|------|------|------|
| 搜索预算 | 代码层 count + 拦截 | 20 次 | 防止无限搜索 |
| 去重缓存 | query hash → 跳过 | 重复 ≥1 次 | 防止同义反复 |
| 全文控制 | web_fetch 全文→/sources/ | 1200 字符 inline | 上下文不膨胀 |
| 软限制 | 累计工具输出警告 | 18K 字符 | 提示收敛 |
| 硬限制 | 累计工具输出拦截 | 30K 字符 | 强制停止搜索 |
| 研究时限 | fast 模式 | 5 分钟 | 保证快速产出 |

### 5.5 知识库 RAG

```
历史研究摘要
  │
  ▼
Summarizer LLM → 分类 + 浓缩
  │
  ▼
BGE-M3 嵌入 → Chroma 向量库（持久化）
  │
  ▼
检索时：
  1. dense recall top-20 (BGE-M3)
  2. sparse recall (lexical weights)
  3. RRF 融合 (Reciprocal Rank Fusion)
  4. Cross-Encoder 重排 → top-3
  5. 父文档去重
```

### 5.6 可审计的运行状态

每次研究独立 Run 目录，支持事后审查和断点恢复：

```
runs/<YYYYMMDD_HHMMSS>/
├── workspace/
│   ├── report.md / report.docx
│   ├── research_summary.txt
│   └── notes/ (Researcher 返回归档)
├── sources/ (来源全文)
└── state/
    ├── events.jsonl      (完整事件流)
    ├── sources.jsonl     (来源账本)
    ├── claims.jsonl      (论断账本)
    └── research_progress.json
```

## 六、效果分析与量化评估

### 6.1 研究 Agent 的评估挑战

研究 Agent 与传统的 QA/分类任务不同，难以用单一的准确率指标衡量。评估维度包括：
- **事实准确性**：报告中的论断是否有来源支撑
- **覆盖完整性**：是否覆盖了课题的关键维度
- **推理深度**：是否进行了跨来源的综合分析
- **引用可追溯性**：每个关键结论能否追溯到原始来源
- **效率**：搜索次数/时间 vs 产出质量的投入产出比

### 6.2 本项目可用的量化指标

| 指标 | 测量方式 | 本项目参考值 |
|------|---------|------------|
| **引用密度** | 报告中引用数 / 报告字数 | V3.0 CSGO 报告: 40 引用 / 23KB ≈ 1.7 引用/KB |
| **来源追溯率** | 有 source_id 的发现 / 总发现 | source_registry 全程追踪，理论 100% |
| **核验覆盖** | 已核验 claims / 总 claims | Verifier 默认最多 20 条 claim |
| **报告结构完整性** | 包含摘要+背景+发现+分析+结论+自评 | Supervisor 强制 7 阶段 |
| **矛盾标注数** | 报告中标注的来源矛盾 | V3.0 CSGO 3 处，V3.1 LLM 报告 3 处 |
| **搜索效率** | 搜索次数 / 报告质量 | fast: 5 次, deep/max: 20 次上限 |
| **可恢复性** | 中断后能否继续 | events.jsonl + --resume 支持 |

### 6.3 业界相关 Benchmark

| Benchmark | 维度 | 适用性 |
|-----------|------|--------|
| **GAIA** (Meta/Fair) | 多步推理 + 工具使用 | ⭐⭐⭐ 最适合，考察 Agent 使用搜索和浏览工具解决多步问题的能力 |
| **WebArena** | Web 环境操作 | ⭐⭐ 可借鉴，评估 Agent 在真实 Web 环境中的导航和搜索 |
| **FRAMES** (Google) | 事实检索与综合 | ⭐⭐⭐ 评估多文档事实检索和综合能力，与本研究场景高度匹配 |
| **Humanity's Last Exam** | 专家级问答 | ⭐ 太难，本研究目标是深度调研不是超人类问答 |
| **MMLU-Pro** | 知识广度 | ⭐ 标准知识评测，不衡量搜索和研究过程 |
| **SimpleQA** (OpenAI) | 事实准确性 | ⭐⭐ 可借鉴其方法论衡量单一事实准确性 |
| **STORM** (Stanford) | 报告生成质量 | ⭐⭐⭐ 最相关——评估从网络搜索生成维基式报告的质量 |

### 6.4 本研究实测定量对比（V3.0 vs V3.1）

同课题"CSGO游戏发展史"，deep 模式：

| 指标 | V3.0 (OODA) | V3.1 (ReAct) | 结论 |
|------|------------|-------------|------|
| 报告大小 | 23KB | 36KB | V3.1 信息量更大 |
| 运行时间 | ~10分钟 | ~34分钟 | V3.0 效率更高 |
| 引用数 | ~40 | ~102 | V3.1 引用更充分 |
| 量化表 | 1张 | 3张 | V3.1 结构化更好 |
| 矛盾标注 | 0 | 3处 | V3.1 批判性更强 |
| 投入产出比 | **更好** | 较差 | V3.0 作为最终版本的原因 |

**选择 V3.0 作为 V4.0 基础的原因**：OODA 循环在保持报告质量（23KB, 40 引用）的同时，效率是 ReAct 的 3.4 倍（10 分钟 vs 34 分钟）。

### 6.5 定性评估维度

- **可审计性**：每条论断 → source_id → 原始文件，完整链路可追溯
- **可恢复性**：events.jsonl 记录全流程，`--resume` 支持断点续跑
- **可扩展性**：config.py 160+ 参数，CLI 全覆盖，新搜索源只需加 @tool
- **信息隔离**：每个 Researcher 独立上下文，搜索结果不互相污染
- **渐进式加载**：Skills 按需注入 Prompt，不增加常态上下文负担
